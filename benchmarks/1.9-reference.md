# 1.9 Cross-Version Performance Reference

**Captured:** 2026-04-17
**Source tag:** `1.9.1.0` (commit 1.9.1.0)
**Hardware:** Local development machine (Windows 11, same session as 2.0 comparison)
**Node:** v22.17.0
**Vitest:** 3.2.4

## What was measured

The 1.9 `getDataRow` function (from `src/lib/dataset/processing.ts` at tag `1.9.1.0`) — the per-row inner-loop function equivalent of 2.0's `buildDataRow`. The function was extracted into a standalone harness in the 2.0 branch with its dependencies resolved against 2.0's `@deneb-viz/data-core` packages (constants and highlight functions exist in both versions).

**Vendored from 1.9:** `getCastedPrimitiveValue` (5-line date-type check), `AugmentedMetadataField` type shape. Everything else was imported from 2.0's packages.

**Drilldown paths were skipped** (`hasDrilldown: false`) since 2.0's `buildDataRow` benchmarks also exclude the extraction preamble, making the comparison tighter.

## 1.9 Results

### With cross-highlight (realistic path)

| Tier | hz (ops/sec) | mean (ms) |
|---|---|---|
| 1,000 rows | 1,531 | 0.65 |
| 10,000 rows | 164 | 6.10 |
| 100,000 rows | 15.2 | 65.9 |

### Without cross-highlight (baseline path)

| Tier | hz (ops/sec) | mean (ms) |
|---|---|---|
| 1,000 rows | 2,461 | 0.41 |
| 10,000 rows | 211 | 4.74 |
| 100,000 rows | 25.9 | 38.6 |

## 2.0 Comparison (same machine, same session)

The closest 2.0 equivalent is `buildDataRow` with the **field-only plan, passthrough provider** (same 2-field composition, highlight enabled on measure, no format/formatted support fields). This is the most apples-to-apples comparison since 1.9 did not have configurable support fields.

| Tier | 1.9 (hz) | 2.0 field-only passthrough (hz) | Δ |
|---|---|---|---|
| 1,000 rows | 1,531 | 1,896 | **2.0 is ~24% faster** |
| 10,000 rows | 164 | 186 | **2.0 is ~13% faster** |
| 100,000 rows | 15.2 | 18.4 | **2.0 is ~21% faster** |

With the **field-only plan, realistic provider** (approximating the cost of actual formatting):

| Tier | 1.9 (hz) | 2.0 field-only realistic (hz) | Δ |
|---|---|---|---|
| 1,000 rows | 1,531 | 1,667 | **2.0 is ~9% faster** |
| 10,000 rows | 164 | 142 | **2.0 is ~13% slower** |
| 100,000 rows | 15.2 | 14.9 | **2.0 is ~2% slower** |

## Interpretation

**2.0 is not slower than 1.9.** The passthrough comparison (closest apples-to-apples) shows 2.0 is 13-24% *faster* at all tiers, indicating the `buildProcessingPlan`/`buildDataRow` refactor improved per-row overhead despite adding the configurable support field system.

The realistic-provider comparison shows mixed results (9% faster at 1K, 13% slower at 10K, 2% slower at 100K) — but this is expected because:
1. The realistic provider does `String(value) + formatString` concatenation and numeric comparison per row — work that 1.9's `getDataRow` does NOT do (1.9 had no per-row formatted/format field emission).
2. The realistic provider simulates NEW functionality in 2.0 (configurable support fields), so the comparison is measuring 2.0 doing more work, not doing the same work slower.

**Conclusion:** The 2.0 refactor maintained or improved baseline throughput for equivalent workloads. The additional cost visible in the realistic-provider comparison is attributable to new features (support field generation), not to a regression in the core processing loop.

## Caveats

- **Hardware parity:** Both 1.9 and 2.0 numbers were captured on the same local machine in the same session. The CI baseline (committed in `benchmarks/baselines/data-core.json`) was captured on `ubuntu-22.04` and has different absolute values. This cross-version comparison should only be used for ratios, not absolute times.
- **Function boundaries differ:** 1.9's `getDataRow` takes a transposed `values[fieldIndex][rowIndex]` array; 2.0's `buildDataRow` takes a per-row `baseValues[fieldIndex]` array. The memory access patterns differ slightly but the algorithmic work is comparable.
- **Drilldown excluded:** Both sides skip drilldown processing. If drilldown performance differs between versions, this comparison does not capture it.
- **No format/formatted in 1.9:** The 1.9 function did not emit format or formatted support fields. The 2.0 passthrough comparison disables these via the plan flags, making it the fairest comparison. The realistic comparison includes them, which inflates the 2.0 cost relative to 1.9.
